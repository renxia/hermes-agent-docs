# 测试驱动开发（TDD）

## 概述

先写测试。观察它失败。编写最少代码使其通过。

**核心原则：** 如果你没看到测试失败，你就不知道它是否测试了正确的东西。

**违反规则的字面意思就是违反规则的精神。**

## 何时使用

**总是：**
- 新功能
- 修复错误
- 重构
- 行为变更

**例外情况（先询问用户）：**
- 一次性原型
- 生成的代码
- 配置文件

想着“就这次跳过TDD”？停下。那是合理化。

## 铁律

```
没有先写一个失败的测试，就不要写生产代码
```

在测试之前写了代码？删掉。从头开始。

**没有例外：**
- 不要保留作为“参考”
- 不要在写测试时“调整”它
- 不要看它
- 删除就是删除

从测试开始新鲜实现。就这样。

## 红-绿-重构循环

### 红色 — 编写失败的测试

编写一个展示应发生什么的最小测试。

**好的测试：**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
名称清晰，测试真实行为，只做一件事。

**坏的测试：**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 重试次数呢？时间呢？
```
名称模糊，测试的是模拟对象而非真实代码。

**要求：**
- 每个测试一个行为
- 清晰的描述性名称（名称里有“和”？拆开它）
- 测试真实代码，而非模拟（除非真正不可避免）
- 名称描述行为，而非实现

### 验证红色 — 观察它失败

**强制性。永远不要跳过。**

```bash
# 使用终端工具运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v
```

确认：
- 测试失败（不是拼写错误导致的错误）
- 失败消息符合预期
- 失败是因为缺少功能

**测试立即通过？** 你在测试现有行为。修复测试。

**测试出错？** 修复错误，重新运行直到它正确地失败。

### 绿色 — 最少代码

编写最简单的代码使测试通过。仅此而已。

**好：**
```python
def add(a, b):
    return a + b  # 没有多余的
```

**坏：**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 多余！
    return result
```

不要添加功能，重构其他代码，或超越测试“改进”。

**在绿色阶段作弊是可以的：**
- 硬编码返回值
- 复制粘贴
- 重复代码
- 跳过边界情况

我们会在重构阶段修复它。

### 验证绿色 — 观察它通过

**强制性。**

```bash
# 运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v

# 然后运行所有测试以检查回归
pytest tests/ -q
```

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（没有错误，警告）

**测试失败？** 修复代码，不是测试。

**其他测试失败？** 立即修复回归。

### 重构 — 清理

仅在绿色之后：
- 消除重复
- 改进名称
- 提取帮助函数
- 简化表达式

在整个过程中保持测试绿色。不要添加行为。

**如果在重构期间测试失败：** 立即撤销。采取更小的步骤。

### 重复

为下一个行为测试下一个失败的测试。一次一个周期。

## 为什么顺序重要

**“我会在之后写测试来验证它有效”**

在代码之后编写的测试会立即通过。立即通过什么也证明不了：
- 可能测试了错误的东西
- 可能测试了实现，而不是行为
- 可能遗漏了你忘记的边界情况
- 你从未看到它捕获错误

先测试迫使你看到测试失败，证明它确实测试了某些东西。

**“我已经手动测试了所有边界情况”**

手动测试是临时的。你以为你测试了一切，但是：
- 没有记录你测试了什么
- 代码更改时无法重新运行
- 在压力下容易忘记情况
- “我试的时候有效了” ≠ 全面

自动化测试是系统的。它们每次都以相同的方式运行。

**“删除 X 小时的工作是浪费”**

沉没成本谬误。时间已经过去了。你现在选择：
- 删除并使用TDD重写（高置信度）
- 保留它并事后添加测试（低置信度，可能有错误）

“浪费”是保留你无法信任的代码。

**“TDD是教条主义，务实意味着适应”**

TDD *是* 务实的：
- 在提交前发现错误（比事后调试更快）
- 防止回归（测试立即捕获破坏）
- 记录行为（测试展示如何使用代码）
- 实现重构（自由更改，测试捕获破坏）

“务实”的捷径 = 在生产中调试 = 更慢。

**“之后的测试达到相同的目标 —— 是精神而非仪式”**

不。事后测试回答“这是做什么的？”先测试回答“这应该做什么？”

事后测试受你的实现影响。你测试的是你构建的东西，而不是要求的东西。先测试迫使在实现之前发现边界情况。

## 常见的合理化

| 借口 | 现实 |
|--------|---------|
| “太简单不用测试” | 简单的代码会出错。测试只需30秒。 |
| “我会之后再测” | 立即通过的测试证明不了什么。 |
| “之后的测试达到相同目标” | 事后测试 = “这是做什么的？”先测试 = “这应该做什么？” |
| “已经手动测试过了” | 临时 ≠ 系统。没有记录，无法重新运行。 |
| “删除X小时是浪费” | 沉没成本谬误。保留未经验证的代码是技术债务。 |
| “保留作为参考，先写测试” | 你会调整它。那是事后测试。删除就是删除。 |
| “需要先探索” | 好的。扔掉探索，从TDD开始。 |
| “测试难 = 设计不清” | 听测试的。难测试 = 难使用。 |
| “TDD会让我慢下来” | TDD比调试更快。务实 = 先测试。 |
| “手动测试更快” | 手动无法证明边界情况。你会在每次更改后重新测试。 |
| “现有代码没有测试” | 你在改进它。为你接触的代码添加测试。 |

## 危险信号 — 停下并重新开始

如果你发现自己在做这些中的任何一件，删除代码并用TDD重新开始：

- 代码先于测试
- 测试在实现之后
- 测试首次运行就通过
- 无法解释为什么测试失败
- 测试“稍后”添加
- 合理化“就这一次”
- “我已经手动测试过了”
- “之后的测试达到相同目的”
- “保留作为参考”或“调整现有代码”
- “已经花了X小时，删除是浪费”
- “TDD是教条，我很务实”
- “这次不同因为……”

**所有这些都意味着：删除代码。用TDD重新开始。**

## 验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有测试
- [ ] 在实现前观察了每个测试失败
- [ ] 每个测试因为预期原因失败（缺少功能，不是拼写错误）
- [ ] 编写了最少的代码使每个测试通过
- [ ] 所有测试通过
- [ ] 输出干净（没有错误，警告）
- [ ] 测试使用真实代码（仅在不可避免时使用模拟）
- [ ] 覆盖了边界情况和错误

无法勾选所有复选框？你跳过了TDD。重新开始。

## 卡住时

| 问题 | 解决方案 |
|---------|----------|
| 不知道如何测试 | 编写期望的API。先写断言。询问用户。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 必须模拟所有东西 | 代码耦合太紧。使用依赖注入。 |
| 测试设置庞大 | 提取帮助函数。仍然复杂？简化设计。 |

## Hermes 智能体集成

### 运行测试

使用 `terminal` 工具在每个步骤运行测试：

```python
# 红色 — 验证失败
terminal("pytest tests/test_feature.py::test_name -v")

# 绿色 — 验证通过
terminal("pytest tests/test_feature.py::test_name -v")

# 完整套件 — 验证没有回归
terminal("pytest tests/ -q")
```

### 使用 delegate_task

当分派子智能体进行实现时，在目标中强制执行TDD：

```python
delegate_task(
    goal="使用严格的TDD实现[功能]",
    context="""
    遵循测试驱动开发技能：
    1. 先写失败的测试
    2. 运行测试以验证它失败
    3. 编写最少的代码使其通过
    4. 运行测试以验证它通过
    5. 如果需要则重构
    6. 提交

    项目测试命令：pytest tests/ -q
    项目结构：[描述相关文件]
    """,
    toolsets=['terminal', 'file']
)
```

### 与 systematic-debugging 结合

发现错误？编写复现它的失败测试。遵循TDD周期。测试证明了修复并防止回归。

永远不要在没有测试的情况下修复错误。

## 测试反模式

- **测试模拟行为而非真实行为** — 模拟应该验证交互，而不是取代被测系统
- **测试实现细节** — 测试行为/结果，而不是内部方法调用
- **只测快乐路径** — 总要测试边界情况、错误和边界
- **脆弱的测试** — 测试应该验证行为，而不是结构；重构不应该破坏它们

## 最终规则

```
生产代码 → 测试存在且首先失败
否则 → 不是TDD
```

没有用户的明确许可，没有例外。
