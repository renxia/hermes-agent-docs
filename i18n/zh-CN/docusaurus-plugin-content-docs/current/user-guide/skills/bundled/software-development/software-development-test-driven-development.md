# 测试驱动开发 (TDD)

## 概述

先编写测试。观察其失败。编写最少的代码使其通过。

**核心原则：** 如果你没有看到测试失败，你就不知道它是否测试了正确的内容。

**违反规则的条文就是违反规则的精神。**

## 何时使用

**始终：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**例外（先询问用户）：**
- 一次性原型
- 生成的代码
- 配置文件

想着“这次跳过 TDD”？停下。那是合理化。

## 铁律

```
没有失败的测试，就没有生产代码
```

在测试之前编写代码？删除它。重新开始。

**没有例外：**
- 不要将其保留为“参考”
- 不要在编写测试时“适配”它
- 不要看它
- 删除就是删除

完全根据测试重新实现。仅此而已。

## 红-绿-重构循环

### 红 — 编写失败的测试

编写一个最小的测试，展示应该发生什么。

**好的测试：**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
名称清晰，测试真实行为，一件事。

**糟糕的测试：**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 重试次数呢？时间呢？
```
名称模糊，测试的是模拟对象而不是真实代码。

**要求：**
- 每个测试一个行为
- 清晰的描述性名称（名称中有“和”？拆分它）
- 真实代码，而不是模拟对象（除非真正不可避免）
- 名称描述行为，而不是实现

### 验证红 — 观察其失败

**强制要求。切勿跳过。**

```bash
# 使用终端工具运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v
```

确认：
- 测试失败（不是因为拼写错误导致的错误）
- 失败消息符合预期
- 因为功能缺失而失败

**测试立即通过？** 你在测试现有行为。修复测试。

**测试出错？** 修复错误，重新运行直到正确失败。

### 绿 — 最少的代码

编写最简单的代码来通过测试。仅此而已。

**好的：**
```python
def add(a, b):
    return a + b  # 没有多余内容
```

**糟糕的：**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 多余！
    return result
```

不要添加功能、重构其他代码或“改进”超出测试范围的内容。

**在绿阶段作弊是可以的：**
- 硬编码返回值
- 复制粘贴
- 重复代码
- 跳过边缘情况

我们将在重构阶段修复它。

### 验证绿 — 观察其通过

**强制要求。**

```bash
# 运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v

# 然后运行所有测试以检查是否有回归
pytest tests/ -q
```

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（没有错误、警告）

**测试失败？** 修复代码，而不是测试。

**其他测试失败？** 立即修复回归。

### 重构 — 清理

仅在绿阶段之后：
- 删除重复
- 改进名称
- 提取辅助函数
- 简化表达式

在整个过程中保持测试为绿色。不要添加行为。

**如果在重构期间测试失败：** 立即撤销。采取更小的步骤。

### 重复

为下一个行为编写下一个失败的测试。一次一个循环。

## 为什么顺序很重要

**“我稍后会编写测试来验证它是否有效”**

在代码之后编写的测试会立即通过。立即通过什么也证明不了：
- 可能测试了错误的内容
- 可能测试了实现，而不是行为
- 可能遗漏了你忘记的边缘情况
- 你从未见过它捕获错误

测试优先迫使你看到测试失败，证明它实际上测试了某些内容。

**“我已经手动测试了所有边缘情况”**

手动测试是临时的。你以为你测试了所有内容，但是：
- 没有你测试内容的记录
- 代码更改时无法重新运行
- 在压力下很容易忘记情况
- “我试过了，它有效” ≠ 全面

自动化测试是系统性的。它们每次都按相同的方式运行。

**“删除 X 小时的工作是浪费”**

沉没成本谬误。时间已经过去了。你现在选择：
- 删除并使用 TDD 重写（高信心）
- 保留它并在之后添加测试（低信心，可能有错误）

“浪费”在于保留你无法信任的代码。

**“TDD 是教条的，务实意味着适应”**

TDD 是务实的：
- 在提交之前发现错误（比之后调试更快）
- 防止回归（测试立即捕获中断）
- 记录行为（测试显示如何使用代码）
- 启用重构（自由更改，测试捕获中断）

“务实”的捷径 = 在生产环境中调试 = 更慢。

**“之后的测试实现了相同的目标 — 这是精神而不是仪式”**

不。之后的测试回答“这是做什么的？”测试优先回答“这应该做什么？”

之后的测试会受到你的实现的影响。你测试的是你构建的内容，而不是所需的内容。测试优先迫使你在实现之前发现边缘情况。

## 常见的合理化

| 借口 | 现实 |
|--------|---------|
| “太简单了，不需要测试” | 简单代码也会出错。测试只需 30 秒。 |
| “我稍后会测试” | 测试立即通过什么也证明不了。 |
| “之后的测试实现了相同的目标” | 之后的测试 = “这是做什么的？”测试优先 = “这应该做什么？” |
| “已经手动测试过了” | 临时 ≠ 系统性。没有记录，无法重新运行。 |
| “删除 X 小时是浪费” | 沉没成本谬误。保留未经验证的代码是技术债务。 |
| “保留为参考，先写测试” | 你会适配它。那就是之后的测试。删除就是删除。 |
| “需要先探索” | 好的。丢弃探索，从 TDD 开始。 |
| “测试困难 = 设计不清晰” | 倾听测试。难以测试 = 难以使用。 |
| “TDD 会让我慢下来” | TDD 比调试更快。务实 = 测试优先。 |
| “手动测试更快” | 手动测试不能证明边缘情况。你每次更改都要重新测试。 |
| “现有代码没有测试” | 你正在改进它。为你接触的代码添加测试。 |

## 危险信号 — 停止并重新开始

如果你发现自己做了以下任何事情，请删除代码并使用 TDD 重新开始：

- 在测试之前编写代码
- 在实现之后编写测试
- 测试在第一次运行时立即通过
- 无法解释测试失败的原因
- 测试是“稍后”添加的
- 将“仅此一次”合理化
- “我已经手动测试过了”
- “之后的测试实现了相同的目的”
- “保留为参考”或“适配现有代码”
- “已经花了 X 小时，删除是浪费”
- “TDD 是教条的，我很务实”
- “这不同，因为……”

**所有这些意味着：删除代码。使用 TDD 重新开始。**

## 验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有一个测试
- [ ] 在实现之前观察每个测试失败
- [ ] 每个测试都因预期原因而失败（功能缺失，而不是拼写错误）
- [ ] 编写了最少的代码来通过每个测试
- [ ] 所有测试都通过
- [ ] 输出干净（没有错误、警告）
- [ ] 测试使用真实代码（仅在不可避免时才使用模拟对象）
- [ ] 边缘情况和错误已涵盖

无法勾选所有框？你跳过了 TDD。重新开始。

## 当遇到困难时

| 问题 | 解决方案 |
|---------|----------|
| 不知道如何测试 | 编写期望的 API。先编写断言。询问用户。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 必须模拟所有内容 | 代码耦合太紧。使用依赖注入。 |
| 测试设置巨大 | 提取辅助函数。仍然复杂？简化设计。 |

## Hermes 智能体集成

### 运行测试

在每个步骤中使用 `terminal` 工具运行测试：

```python
# 红 — 验证失败
terminal("pytest tests/test_feature.py::test_name -v")

# 绿 — 验证通过
terminal("pytest tests/test_feature.py::test_name -v")

# 完整套件 — 验证没有回归
terminal("pytest tests/ -q")
```

### 与 delegate_task 一起使用

在分派子智能体进行实现时，在目标中强制执行 TDD：

```python
delegate_task(
    goal="使用严格的 TDD 实现 [功能]",
    context="""
    遵循测试驱动开发技能：
    1. 先编写失败的测试
    2. 运行测试以验证其失败
    3. 编写最少的代码以通过
    4. 运行测试以验证其通过
    5. 如果需要，进行重构
    6. 提交

    项目测试命令：pytest tests/ -q
    项目结构：[描述相关文件]
    """,
    toolsets=['terminal', 'file']
)
```

### 与 systematic-debugging 一起使用

发现错误？编写失败的测试来重现它。遵循 TDD 循环。测试证明修复并防止回归。

永远不要在没有测试的情况下修复错误。
## 测试反模式

- **测试模拟行为而不是真实行为** — 模拟对象应验证交互，而不是替换被测系统
- **测试实现细节** — 测试行为/结果，而不是内部方法调用
- **仅测试快乐路径** — 始终测试边缘情况、错误和边界
- **脆弱的测试** — 测试应验证行为，而不是结构；重构不应破坏它们

## 最终规则

```
生产代码 → 测试存在并且首先失败
否则 → 不是 TDD
```

未经用户明确许可，不得有任何例外。
