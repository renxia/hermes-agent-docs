# 系统化调试

## 概述

随机修复浪费时间并会引入新错误。快速补丁只能掩盖根本问题。

**核心原则：** 在尝试修复之前，**务必**找到根本原因。仅修复症状等于失败。

**违反此流程的字面要求，就是违反调试的精神。**

## 铁律

```
未进行根本原因调查，不得进行任何修复
```

如果您尚未完成第1阶段，则不能提出任何修复方案。

## 何时使用

适用于**任何**技术问题：
- 测试失败
- 生产环境中的错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其适用于以下情况：**
- 时间紧迫（紧急情况会让人产生猜测的冲动）
- “只需一个快速修复”看似显而易见
- 您已经尝试过多种修复方案
- 之前的修复无效
- 您尚未完全理解问题

**切勿跳过以下情况：**
- 问题看似简单（简单错误也有其根本原因）
- 您很匆忙（匆忙只会导致返工）
- 有人要求立即修复（系统化方法比反复试错更快）

## 四个阶段

您**必须**完成每个阶段才能进入下一阶段。

---

## 第1阶段：根本原因调查

**在尝试任何修复之前：**

### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 它们通常包含确切的解决方案
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误代码

**操作：** 对相关源文件使用 `read_file`。使用 `search_files` 在代码库中搜索错误字符串。

### 2. 稳定复现问题

- 能否可靠地触发该问题？
- 确切步骤是什么？
- 是否每次都会发生？
- 如果无法复现 → 收集更多数据，不要猜测

**操作：** 使用 `terminal` 工具运行失败的测试或触发错误：

```bash
# 运行特定失败的测试
pytest tests/test_module.py::test_name -v

# 使用详细输出运行
pytest tests/test_module.py -v --tb=long
```

### 3. 检查近期变更

- 哪些变更可能导致此问题？
- Git diff、近期提交
- 新依赖项、配置变更

**操作：**

```bash
# 近期提交
git log --oneline -10

# 未提交的变更
git diff

# 特定文件的变更
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多组件系统中收集证据

**当系统包含多个组件时（API → 服务 → 数据库，CI → 构建 → 部署）：**

**在提出修复方案之前，添加诊断性检测：**

对每个组件边界：
- 记录进入组件的数据
- 记录离开组件的数据
- 验证环境/配置传播
- 检查每一层的状态

运行一次以收集证据，明确问题**出现在何处**。
然后分析证据以识别出故障组件。
再对该特定组件进行调查。

### 5. 追踪数据流

**当错误深埋在调用堆栈中时：**

- 错误值源自何处？
- 是什么函数以错误值调用了此函数？
- 持续向上游追踪，直到找到源头
- 在源头修复，而非在症状处修复

**操作：** 使用 `search_files` 追踪引用：

```python
# 查找调用该函数的位置
search_files("function_name(", path="src/", file_glob="*.py")

# 查找设置该变量的位置
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 第1阶段完成检查清单

- [ ] 错误信息已完整阅读并理解
- [ ] 问题已稳定复现
- [ ] 近期变更已识别并审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题已隔离至特定组件/代码
- [ ] 已形成根本原因假设

**停止：** 在理解**为何**发生之前，不要进入第2阶段。

---

## 第2阶段：模式分析

**在修复之前找到模式：**

### 1. 找到正常工作的示例

- 定位同一代码库中类似的正常工作代码
- 什么代码与出问题的代码相似但能正常工作？

**操作：** 使用 `search_files` 查找可比模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 与参考实现对比

- 如果正在实现某种模式，请**完整阅读**参考实现
- 不要略读 — 逐行阅读
- 在应用之前充分理解该模式

### 3. 识别差异

- 正常工作与出问题的代码之间有何不同？
- 列出所有差异，无论多小
- 不要假设“那无关紧要”

### 4. 理解依赖关系

- 这需要哪些其他组件？
- 需要哪些设置、配置、环境？
- 它做了哪些假设？

---

## 第3阶段：假设与测试

**科学方法：**

### 1. 提出单一假设

- 清晰陈述：“我认为 X 是根本原因，因为 Y”
- 写下来
- 具体而非模糊

### 2. 最小化测试

- 做出**尽可能小**的变更来测试假设
- 一次只改变一个变量
- 不要同时修复多个问题

### 3. 验证后再继续

- 是否有效？→ 进入第4阶段
- 无效？→ 提出**新**假设
- **不要**在此基础上添加更多修复

### 4. 当您不知道时

- 说“我不理解 X”
- 不要假装知道
- 向用户寻求帮助
- 进一步研究

---

## 第4阶段：实施

**修复根本原因，而非症状：**

### 1. 创建失败的测试用例

- 最简单的复现方式
- 如果可能，使用自动化测试
- 修复前**必须**具备
- 使用 `test-driven-development` 技能

### 2. 实施单一修复

- 针对已识别的根本原因
- 一次只做**一个**变更
- 不要做“既然来了就顺便”的改进
- 不要捆绑重构

### 3. 验证修复

```bash
# 运行特定的回归测试
pytest tests/test_module.py::test_regression -v

# 运行完整套件 — 无回归
pytest tests/ -q
```

### 4. 如果修复无效 — 三法则

- **停止。**
- 计数：您已尝试了多少次修复？
- 如果 &lt; 3：返回第1阶段，结合新信息重新分析
- **如果 ≥ 3：停止并质疑架构（见下文第5步）**
- 未经架构讨论，**不得**尝试第4次修复

### 5. 如果3次以上修复失败：质疑架构

**表明存在架构问题的模式：**
- 每次修复都暴露出不同位置的新共享状态/耦合
- 修复需要“大规模重构”才能实现
- 每次修复都在其他地方引发新的症状

**停止并质疑基本原理：**
- 此模式从根本上是否合理？
- 我们是否“仅凭惯性坚持下去”？
- 我们应该重构架构，还是继续修复症状？

**在尝试更多修复之前，与用户讨论。**

这不是失败的假设 — 这是错误的架构。

---

## 危险信号 — 停止并遵循流程

如果您发现自己有以下想法：
- “现在先快速修复，以后再调查”
- “只需尝试更改 X 看看是否有效”
- “添加多个变更，运行测试”
- “跳过测试，我将手动验证”
- “很可能是 X，让我修复它”
- “我不完全理解，但这可能有效”
- “模式说是 X，但我会以不同方式适配”
- “以下是主要问题：[列出未经调查的修复方案]”
- 在追踪数据流之前就提出解决方案
- **“再试一次修复”（当已尝试2次以上时）**
- **每次修复都在不同位置暴露出新问题**

**以上所有情况都意味着：停止。返回第1阶段。**

**如果3次以上修复失败：** 质疑架构（第4阶段第5步）。

## 常见合理化借口

| 借口 | 现实 |
|--------|---------|
| “问题很简单，不需要流程” | 简单问题也有根本原因。对于简单错误，流程同样高效。 |
| “紧急情况，没时间走流程” | 系统化调试比反复试错**更快**。 |
| “先试试这个，然后再调查” | 第一次修复就设定了模式。从一开始就做对。 |
| “我会在确认修复有效后再写测试” | 未经测试的修复无法持久。先写测试才能证明其有效性。 |
| “一次修复多个问题可以节省时间” | 无法隔离哪些修复有效。会引发新错误。 |
| “参考太长，我会适配该模式” | 部分理解必然导致错误。请完整阅读。 |
| “我看到问题了，让我修复它” | 看到症状 ≠ 理解根本原因。 |
| “再试一次修复”（在2次失败后） | 3次以上失败 = 架构问题。质疑该模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根本原因** | 阅读错误、复现、检查变更、收集证据、追踪数据流 | 理解**什么**和**为什么** |
| **2. 模式** | 找到正常示例、对比、识别差异 | 知道有何不同 |
| **3. 假设** | 形成理论、最小化测试、一次一个变量 | 已确认或提出新假设 |
| **4. 实施** | 创建回归测试、修复根本原因、验证 | 错误已解决，所有测试通过 |

## Hermes 智能体集成

### 调查工具

在阶段 1 期间使用这些 Hermes 工具：

- **`search_files`** — 查找错误字符串、跟踪函数调用、定位模式
- **`read_file`** — 读取带行号的源代码以进行精确分析
- **`terminal`** — 运行测试、检查 git 历史、复现 bug
- **`web_search`/`web_extract`** — 研究错误信息、库文档

### 使用 delegate_task

对于复杂的多组件调试，派遣调查子智能体：

```python
delegate_task(
    goal="调查为什么 [特定测试/行为] 失败",
    context="""
    遵循系统化调试技能：
    1. 仔细阅读错误信息
    2. 复现问题
    3. 跟踪数据流以找到根本原因
    4. 报告发现 — 暂时不要修复

    错误：[粘贴完整错误]
    文件：[失败代码的路径]
    测试命令：[精确命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 使用测试驱动开发

修复 bug 时：
1. 编写一个能复现 bug 的测试（红色）
2. 系统化调试以找到根本原因
3. 修复根本原因（绿色）
4. 测试证明修复有效并防止回归

## 实际影响

来自调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时反复尝试
- 首次修复成功率：95% 对比 40%
- 引入新 bug：接近零 对比 常见

**没有捷径。不要猜测。系统化方法永远胜出。**
