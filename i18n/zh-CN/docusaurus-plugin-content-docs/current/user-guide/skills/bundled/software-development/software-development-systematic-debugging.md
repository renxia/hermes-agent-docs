# 系统性调试

## 概述

随机的修复会浪费时间并制造新的错误。快速修补掩盖了根本问题。

**核心原则：** 在尝试修复之前，必须先找到根本原因。只修复症状是失败的。

**违反此流程的字面意思就是违背调试的精神。**

## 铁律

```
在调查根本原因之前，不要进行任何修复
```

如果你尚未完成阶段一，就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境中的错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间紧迫（紧急情况让人忍不住猜测）
- "一个快速修复"看起来很明显
- 你已经尝试了多种修复方法
- 之前的修复没有起作用
- 你并不完全理解这个问题

**不要跳过的情况：**
- 问题看起来简单（简单的错误也有根本原因）
- 你很匆忙（仓促必然导致返工）
- 有人要求立刻修复（有条不紊比来回折腾更快）

## 四个阶段

你必须在进入下一阶段前完成每个阶段。

---

## 阶段一：根本原因调查

**在尝试任何修复之前：**

### 1. 仔细阅读错误信息

- 不要跳过错误或警告信息
- 它们通常包含确切的解决方案
- 完整地阅读堆栈跟踪
- 记下行号、文件路径、错误代码

**操作：** 对相关源文件使用 `read_file`。使用 `search_files` 在代码库中查找错误字符串。

### 2. 一致地重现

- 你能可靠地触发它吗？
- 确切的步骤是什么？
- 每次都会发生吗？
- 如果无法重现 → 收集更多数据，不要猜测

**操作：** 使用 `terminal` 工具运行失败的测试或触发错误：

```bash
# 运行特定的失败测试
pytest tests/test_module.py::test_name -v

# 运行并显示详细输出
pytest tests/test_module.py -v --tb=long
```

### 3. 检查近期更改

- 有什么更改可能导致了这个问题？
- Git diff，最近的提交
- 新的依赖项，配置更改

**操作：**

```bash
# 最近的提交
git log --oneline -10

# 未提交的更改
git diff

# 特定文件中的更改
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多组件系统中收集证据

**当系统有多个组件时（API -> 服务 -> 数据库，CI -> 构建 -> 部署）：**

**在提出修复方案之前，添加诊断性插桩：**

对于每个组件边界：
- 记录进入组件的数据
- 记录离开组件的数据
- 验证环境/配置的传播
- 检查每一层的状态

运行一次以收集证据，显示问题出在哪里。
然后分析证据以识别故障组件。
接着调查该特定组件。

### 5. 追踪数据流

**当错误深埋在调用栈中时：**

- 错误值源自哪里？
- 是什么用错误值调用了这个函数？
- 持续向上游追踪直到找到源头
- 在源头修复，而不是在症状处修复

**操作：** 使用 `search_files` 追踪引用：

```python
# 查找函数在哪里被调用
search_files("function_name(", path="src/", file_glob="*.py")

# 查找变量在哪里被设置
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 阶段一完成检查清单

- [ ] 错误信息已完全阅读和理解
- [ ] 问题已一致地重现
- [ ] 近期更改已识别并审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题已隔离到特定组件/代码
- [ ] 已形成根本原因假设

**停止：** 在理解问题发生的原因之前，不要进入阶段二。

---

## 阶段二：模式分析

**在修复前找到模式：**

### 1. 查找有效的示例

- 在相同代码库中定位类似的正常工作的代码
- 有什么与出错的东西相似但有效的代码？

**操作：** 使用 `search_files` 查找可比较的模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 与参考进行比较

- 如果正在实现某个模式，请完整阅读参考实现
- 不要略读——阅读每一行
- 在应用之前完全理解该模式

### 3. 识别差异

- 有效和无效的代码之间有什么不同？
- 列出每一个不同之处，无论多么微小
- 不要假设"这不重要"

### 4. 理解依赖关系

- 这个组件还需要其他什么？
- 需要什么设置、配置、环境？
- 它做了什么假设？

---

## 阶段三：假设与测试

**科学方法：**

### 1. 形成一个单一假设

- 明确陈述："我认为X是根本原因，因为Y"
- 写下来
- 具体，而非模糊

### 2. 最小化测试

- 做尽可能小的更改来测试假设
- 一次只改变一个变量
- 不要一次修复多个问题

### 3. 在继续前验证

- 有效吗？ → 阶段四
- 无效？ → 形成新的假设
- 不要在上面叠加更多修复

### 4. 当你不知道时

- 说"我不理解X"
- 不要假装知道
- 向用户寻求帮助
- 进行更多研究

---

## 阶段四：实施

**修复根本原因，而不是症状：**

### 1. 创建失败的测试用例

- 尽可能简单的重现方式
- 如果可能，使用自动化测试
- 在修复前必须拥有此测试
- 使用 `test-driven-development` 技能

### 2. 实施单一修复

- 解决已识别的根本原因
- 一次只做一个更改
- 不做"顺便"的改进
- 不进行捆绑重构

### 3. 验证修复

```bash
# 运行特定的回归测试
pytest tests/test_module.py::test_regression -v

# 运行全部测试套件——确保没有回归
pytest tests/ -q
```

### 4. 如果修复不起作用——三次尝试规则

- **停止。**
- 计数：你尝试了多少次修复？
- 如果 < 3次：返回阶段一，用新信息重新分析
- **如果 ≥ 3次：停止并质疑架构（下面的步骤5）**
- 不要在未进行架构讨论的情况下尝试第4次修复

### 5. 如果3次以上修复失败：质疑架构

**指示存在架构问题的模式：**
- 每次修复都在不同位置揭示了新的共享状态/耦合
- 修复需要"大规模重构"才能实施
- 每次修复都在别处制造新的症状

**停止并质疑基本面：**
- 这个模式从根本上是合理的吗？
- 我们是否只是"纯粹出于惯性坚持它"？
- 我们应该重构架构，还是继续修复症状？

**在尝试更多修复之前与用户讨论。**

这不是失败的假设——这是错误的架构。

---

## 红旗警告——停止并遵循流程

如果你发现自己在想：
- "现在先快速修复，以后再调查"
- "先试着改X看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是X，让我修复它"
- "我不完全理解，但这可能有效"
- "模式说X，但我会以不同方式调整"
- "主要问题是：[列出修复而未调查]"
- 在追踪数据流之前就提出解决方案
- **"再试一次修复"（当已经尝试了2次以上）**
- **每次修复都在不同位置揭示新问题**

**所有这些都意味着：停止。返回阶段一。**

**如果3次以上修复失败：** 质疑架构（阶段四步骤5）。

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单的问题也有根本原因。流程对简单错误来说也很快。 |
| "紧急情况，没时间走流程" | 系统性调试比猜测和尝试的来回折腾更快。 |
| "先试试这个，然后再调查" | 第一次修复设定了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未测试的修复不会持久。先测试才能证明。 |
| "同时进行多个修复节省时间" | 无法隔离哪些有效。会导致新的错误。 |
| "参考太长，我会调整模式" | 部分理解必然导致错误。完整阅读它。 |
| "我看到问题了，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再试一次修复"（2次失败后） | 3次以上失败 = 架构问题。质疑模式，不要再次修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根本原因** | 阅读错误、重现、检查更改、收集证据、追踪数据流 | 理解"是什么"和"为什么" |
| **2. 模式** | 找到有效示例、比较、识别差异 | 知道不同之处 |
| **3. 假设** | 形成理论、最小化测试、一次一个变量 | 假设被确认或需要新假设 |
| **4. 实施** | 创建回归测试、修复根本原因、验证 | 错误解决，所有测试通过 |

## Hermes 智能体集成

### 调查工具

在第一阶段使用这些 Hermes 工具：

- **`search_files`** — 查找错误字符串、追踪函数调用、定位模式
- **`read_file`** — 读取带行号的源代码以便精确分析
- **`terminal`** — 运行测试、检查 git 历史、复现 bug
- **`web_search`/`web_extract`** — 研究错误信息、库文档

### 使用 `delegate_task`

对于复杂的多组件调试，可以派遣调查子智能体：

```python
delegate_task(
    goal="调查为什么 [特定测试/行为] 失败",
    context="""
    遵循系统化调试技能：
    1. 仔细阅读错误信息
    2. 复现问题
    3. 追踪数据流以找到根本原因
    4. 报告发现 —— **不要** 修复

    错误：[粘贴完整错误信息]
    文件：[失败代码的路径]
    测试命令：[准确的命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 使用测试驱动开发

修复 bug 时：
1. 编写一个能复现该 bug 的测试（RED）
2. 系统性地调试以找到根本原因
3. 修复根本原因（GREEN）
4. 测试证明修复有效并防止回归

## 实际影响

根据调试会话的经验：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时的反复尝试
- 首次修复率：95% 对比 40%
- 引入新 bug：接近零对比常见

**没有捷径。没有猜测。系统化方法总能赢。**
