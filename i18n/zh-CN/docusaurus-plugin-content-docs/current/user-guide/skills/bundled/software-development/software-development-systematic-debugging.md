# 系统化调试

## 概述

随机修复浪费时间并产生新的错误。快速补丁掩盖了根本问题。

**核心原则：** 在尝试修复之前，始终找到根本原因。症状修复等于失败。

**违反此流程的字面要求，就是违反调试的精神。**

## 铁律

```
在首先进行根本原因调查之前，不得进行任何修复
```

如果您尚未完成阶段 1，则不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境中的错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况下使用此技能：**
- 时间紧迫（紧急情况会让人产生猜测的冲动）
- “只是一个快速修复”看起来显而易见
- 您已经尝试了多种修复方案
- 之前的修复没有奏效
- 您不完全理解问题

**切勿跳过以下情况：**
- 问题看起来简单（简单错误也有根本原因）
- 您赶时间（匆忙保证会导致返工）
- 有人要求立即修复（系统化比反复试错更快）

## 四个阶段

您必须完成每个阶段才能进入下一阶段。

## 阶段 1：根本原因调查

**在尝试任何修复之前：**

### 1. 仔细阅读错误消息

- 不要跳过错误或警告
- 它们通常包含确切的解决方案
- 完整阅读堆栈跟踪
- 注意行号、文件路径、错误代码

**操作：** 对相关源文件使用 `read_file`。使用 `search_files` 在代码库中查找错误字符串。

### 2. 一致地复现问题

- 您能否可靠地触发它？
- 确切的步骤是什么？
- 是否每次都发生？
- 如果无法复现 → 收集更多数据，不要猜测

**操作：** 使用 `terminal` 工具运行失败的测试或触发错误：

```bash
# 运行特定失败的测试
pytest tests/test_module.py::test_name -v

# 使用详细输出运行
pytest tests/test_module.py -v --tb=long
```

### 3. 检查最近的更改

- 什么更改可能导致此问题？
- Git 差异、最近的提交
- 新的依赖项、配置更改

**操作：**

```bash
# 最近的提交
git log --oneline -10

# 未提交的更改
git diff

# 特定文件中的更改
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多组件系统中收集证据

**当系统有多个组件时（API → 服务 → 数据库，CI → 构建 → 部署）：**

**在提出修复方案之前，添加诊断工具：**

对于每个组件边界：
- 记录进入组件的数据
- 记录离开组件的数据
- 验证环境/配置传播
- 检查每一层的状态

运行一次以收集证据，显示问题出现在何处。
然后分析证据以识别失败的组件。
然后调查该特定组件。

### 5. 跟踪数据流

**当错误深入调用堆栈时：**

- 错误值源自何处？
- 什么函数使用错误值调用了此函数？
- 持续向上游跟踪，直到找到源头
- 在源头修复，而不是在症状处修复

**操作：** 使用 `search_files` 跟踪引用：

```python
# 查找函数被调用的位置
search_files("function_name(", path="src/", file_glob="*.py")

# 查找变量被设置的位置
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 阶段 1 完成清单

- [ ] 错误消息已完全阅读并理解
- [ ] 问题已一致地复现
- [ ] 最近的更改已识别并审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题已隔离到特定组件/代码
- [ ] 已形成根本原因假设

**停止：** 在您理解问题发生的原因之前，不要进入阶段 2。

---

## 阶段 2：模式分析

**在修复之前找到模式：**

### 1. 找到工作示例

- 在相同代码库中找到类似的工作代码
- 什么工作代码与损坏的代码相似？

**操作：** 使用 `search_files` 查找可比较的模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 与参考进行比较

- 如果实现一个模式，请完全阅读参考实现
- 不要略读 — 逐行阅读
- 在应用之前完全理解该模式

### 3. 识别差异

- 工作代码和损坏代码之间有什么不同？
- 列出每一个差异，无论多小
- 不要假设“这无关紧要”

### 4. 理解依赖关系

- 这需要哪些其他组件？
- 需要哪些设置、配置、环境？
- 它做了哪些假设？

---

## 阶段 3：假设和测试

**科学方法：**

### 1. 形成一个单一假设

- 清楚地说明：“我认为 X 是根本原因，因为 Y”
- 写下来
- 具体，而不是模糊

### 2. 最小化测试

- 做出尽可能小的更改来测试假设
- 一次只改变一个变量
- 不要一次修复多个问题

### 3. 在继续之前验证

- 它有效吗？→ 阶段 4
- 无效？→ 形成新的假设
- 不要在顶部添加更多修复

### 4. 当您不知道时

- 说“我不理解 X”
- 不要假装知道
- 向用户寻求帮助
- 进行更多研究

---

## 阶段 4：实施

**修复根本原因，而不是症状：**

### 1. 创建失败的测试用例

- 最简单的可能复现
- 如果可能，使用自动化测试
- 修复前必须有
- 使用 `test-driven-development` 技能

### 2. 实施单一修复

- 解决已识别的根本原因
- 一次只进行一次更改
- 不要进行“顺便”的改进
- 不要捆绑重构

### 3. 验证修复

```bash
# 运行特定的回归测试
pytest tests/test_module.py::test_regression -v

# 运行完整套件 — 无回归
pytest tests/ -q
```

### 4. 如果修复无效 — 三次规则

- **停止。**
- 计数：您尝试了多少次修复？
- 如果 < 3：返回阶段 1，用新信息重新分析
- **如果 ≥ 3：停止并质疑架构（见下文步骤 5）**
- 在没有架构讨论的情况下，不要尝试第 4 次修复

### 5. 如果 3+ 次修复失败：质疑架构

**表明架构问题的模式：**
- 每次修复都揭示不同位置的新共享状态/耦合
- 修复需要“大规模重构”才能实现
- 每次修复在其他地方都会产生新的症状

**停止并质疑基本原理：**
- 这个模式从根本上可靠吗？
- 我们是否“仅仅因为惯性而坚持下去”？
- 我们应该重构架构，而不是继续修复症状？

**在尝试更多修复之前，与用户讨论。**

这不是失败的假设 — 这是错误的架构。

---

## 危险信号 — 停止并遵循流程

如果您发现自己想到：
- “现在快速修复，稍后调查”
- “只是尝试更改 X 看看是否有效”
- “添加多个更改，运行测试”
- “跳过测试，我将手动验证”
- “可能是 X，让我修复它”
- “我不完全理解，但这可能有效”
- “模式说 X，但我会以不同方式适应它”
- “以下是主要问题：[列出修复方案而不进行调查]”
- 在跟踪数据流之前提出解决方案
- **“再尝试一次修复”（当已经尝试过 2+ 次时）**
- **每次修复都揭示不同位置的新问题**

**所有这些意味着：停止。返回阶段 1。**

**如果 3+ 次修复失败：** 质疑架构（阶段 4 步骤 5）。

## 常见的合理化借口

| 借口 | 现实 |
|--------|---------|
| “问题简单，不需要流程” | 简单问题也有根本原因。流程对简单错误来说很快。 |
| “紧急情况，没有时间走流程” | 系统化调试比猜测和检查的反复试错更快。 |
| “先试试这个，然后再调查” | 第一次修复就设定了模式。从一开始就做对。 |
| “我会在确认修复有效后再写测试” | 未经测试的修复不会持久。先测试证明它。 |
| “一次进行多个修复可以节省时间” | 无法隔离什么有效。会导致新的错误。 |
| “参考太长，我会适应这个模式” | 部分理解保证会产生错误。完整阅读它。 |
| “我看到问题了，让我修复它” | 看到症状 ≠ 理解根本原因。 |
| “再尝试一次修复”（在 2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根本原因** | 阅读错误、复现、检查更改、收集证据、跟踪数据流 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、比较、识别差异 | 知道有什么不同 |
| **3. 假设** | 形成理论、最小化测试、一次一个变量 | 确认或新的假设 |
| **4. 实施** | 创建回归测试、修复根本原因、验证 | 错误解决，所有测试通过 |

## Hermes 智能体集成

### 调查工具

在阶段 1 期间使用这些 Hermes 工具：

- **`search_files`** — 查找错误字符串、跟踪函数调用、定位模式
- **`read_file`** — 带行号阅读源代码以进行精确分析
- **`terminal`** — 运行测试、检查 git 历史、复现错误
- **`web_search`/`web_extract`** — 研究错误消息、库文档

### 使用 delegate_task

对于复杂的多组件调试，分派调查子智能体：

```python
delegate_task(
    goal="调查为什么 [特定测试/行为] 失败",
    context="""
    遵循系统化调试技能：
    1. 仔细阅读错误消息
    2. 复现问题
    3. 跟踪数据流以找到根本原因
    4. 报告发现 — 不要修复

    错误：[粘贴完整错误]
    文件：[失败代码的路径]
    测试命令：[确切命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 使用 test-driven-development

修复错误时：
1. 编写一个复现错误的测试（红色）
2. 系统化调试以找到根本原因
3. 修复根本原因（绿色）
4. 测试证明修复并防止回归

## 实际影响

根据调试经验：
- 系统性方法：15-30 分钟修复
- 随机修复方法：2-3 小时反复试错
- 首次修复成功率：95% 对比 40%
- 引入新错误：几乎为零 对比 常见

**没有捷径。不要猜测。系统性方法永远胜出。**
