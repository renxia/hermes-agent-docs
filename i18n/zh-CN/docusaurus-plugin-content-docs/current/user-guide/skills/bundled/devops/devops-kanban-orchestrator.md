# 看板编排器 — 分解剧本

> **核心工作流程生命周期**（包括 `kanban_create` 扇出模式和"分解，而非执行"规则）通过 `KANBAN_GUIDANCE` 系统提示块自动注入到每个看板流程中。当您作为整个职责是路由的编排器配置文件时，此技能是更深入的剧本。

## 配置文件由用户配置 — 不是固定的花名册

Hermes 的设置千差万别。有些用户运行一个执行所有任务的配置文件；有些运行一个小队列（`docker-worker`、`cron-worker`）；有些运行一个他们自己命名的精心策划的专家团队。**没有默认的专家名册** — 编排器技能不知道此机器上存在哪些配置文件。

在扇出之前，您必须将分解基于实际存在的配置文件。调度器会静默失败，无法生成未知的被分配者名称 — 它不会自动更正、不会建议、不会回退。因此，在只设置有 `docker-worker` 的环境中，分配给 `researcher` 的卡片会永远停留在 `ready` 状态。

**步骤 0：在规划前发现可用的配置文件。**

使用以下方法之一：

- `hermes profile list` — 打印此机器上配置的配置文件表。如果您有终端工具，则通过该工具运行；否则询问用户。
- `kanban_list(assignee="<some-name>")` — 检查单个名称。对于未知的被分配者，返回空列表（而不是错误），因此这仅能确认您已经考虑过的名称。
- **直接询问用户。** "您设置有哪些配置文件？"是当目标需要多个专家时一个很好的第一步。

将结果缓存到工作记忆中以供对话其余部分使用。每轮重新询问会浪费工具调用。

## 何时使用看板（与直接执行工作）

当以下任何一项为真时，创建看板任务：

1. **需要多个专家。** 研究 + 分析 + 写作是三个配置文件。
2. **工作应在崩溃或重启后保留。** 长时间运行、重复性或重要的。
3. **用户可能希望介入。** 任何步骤的人机协作。
4. **多个子任务可以并行运行。** 扇出以提高速度。
5. **预期需要审查/迭代。** 审查者配置文件对草稿输出进行循环。
6. **审计跟踪很重要。** 看板行永久保存在 SQLite 中。

如果*以上均不适用* — 这是一个一次性的小推理任务 — 请改用 `delegate_task` 或直接回答用户。

## 反诱惑规则

您的职位描述是"路由，而非执行"。执行这些规则的规则：

- **不要自己执行工作。** 您受限的工具集通常甚至不包括用于实现的终端/文件/代码/网络。如果您发现自己"快速修复这个" — 停止并为合适的专家创建任务。
- **对于任何具体任务，创建看板任务并进行分配。** 每次都如此。
- **在创建卡片之前拆分多通道请求。** 一个用户提示可能包含多个独立的工作流。首先提取这些通道，然后为每个通道创建一张卡片，而不是将不相关的工作捆绑到单个实施者卡片中。
- **并行运行独立通道。** 如果两张卡片不需要彼此的输出，请将它们解除链接，以便调度器可以将它们扇出。仅链接真正的数据依赖关系。
- **如果没有适合可用配置文件的专家，请询问用户要创建哪个配置文件或使用哪个现有配置文件。** 不要编造配置文件名称；调度器会静默丢弃未知的被分配者。
- **分解、路由、总结 — 这就是全部工作。**

## 分解剧本

### 步骤 1 — 理解目标

如果目标不明确，请提出澄清性问题。询问成本低；生成错误的队列成本高。

### 步骤 2 — 构思任务图

在创建任何内容之前，大声起草图表（在您对用户的回复中）。将每个具体工作流视为候选卡片：

1. 从请求中提取通道。
2. 将每个通道映射到您在步骤 0 中发现的配置文件之一。如果某个通道不适合任何现有配置文件，请询问用户要使用或创建哪一个。
3. 决定每个通道是独立的还是由另一个通道门控。
4. 创建没有父链接的独立通道作为并行卡片。
5. 创建具有父链接到它们所依赖的通道的综合/审查/集成卡片。

应该扇出的提示示例（使用占位配置文件名称 — 替换为用户设置中存在的任何内容）：

- "构建应用程序" → 一张卡片给设计导向的配置文件用于产品/UI 方向，一到两张卡片给工程配置文件用于实施，如果用户有审查者配置文件，则加上后期的集成/审查卡片。
- "修复阻碍并检查模型变体" → 一张实施卡片用于阻碍修复，加上一张发现/研究卡片用于配置/源代码验证。最终审查者卡片可以依赖于两者。
- "研究文档并实施" → 文档研究卡片可以与代码库发现卡片并行运行；实施仅在确实需要这些发现时才等待。
- "分析此屏幕截图并找到相关代码" → 一张卡片给具有视觉能力的配置文件用于视觉分析，同时另一张搜索代码库。

像"另外"、"最后"或"和"这样的词并不自动意味着依赖关系。它们通常意味着"确保在报告之前覆盖这一点"。仅当一张卡片无法开始直到另一张卡片的输出存在时，才链接任务。

在创建卡片之前向用户展示图表。让他们纠正它 — 包括每个通道应使用哪个实际的配置文件名称。

### 步骤 3 — 创建任务并链接

使用步骤 0 中的配置文件名称。下面的示例使用占位符 `<profile-A>`、`<profile-B>`、`<profile-C>` — 将它们替换为用户实际拥有的配置文件。

```python
t1 = kanban_create(
    title="研究：Postgres 成本与当前对比",
    assignee="<profile-A>",  # 此设置中负责研究的配置文件
    body="比较 3 年时间窗口内的预估基础设施成本、迁移成本和持续运营成本。来源：AWS/GCP 定价、团队时间估算、来自同行的当前 Postgres 账单。",
    tenant=os.environ.get("HERMES_TENANT"),
)["task_id"]

t2 = kanban_create(
    title="研究：Postgres 性能与当前对比",
    assignee="<profile-A>",  # 同一配置文件，并行运行
    body="在我们的预期数据量（约 500GB，峰值 10k QPS）下比较查询延迟、吞吐量和扩展特性。来源：基准论文、公开案例研究、如果容易则进行 pgbench 测试。",
)["task_id"]

t3 = kanban_create(
    title="综合迁移建议",
    assignee="<profile-B>",  # 负责综合/分析的配置文件
    body="阅读 T1（成本）和 T2（性能）的研究结果。生成一份 1 页的建议，包含明确的权衡和通过/不通过的决定。",
    parents=[t1, t2],
)["task_id"]

t4 = kanban_create(
    title="起草决策备忘录",
    assignee="<profile-C>",  # 起草面向用户的散文的配置文件
    body="将分析师的建议转化为一份 2 页的 CTO 备忘录。匹配团队知识库中先前决策备忘录的语气。",
    parents=[t3],
)["task_id"]
```

`parents=[...]` 进行门控提升 — 子任务保持在 `todo` 状态，直到每个父任务达到 `done` 状态，然后自动提升到 `ready`。不需要手动协调；调度器和依赖引擎会处理它。

### 步骤 4 — 完成您自己的任务

如果您是作为任务生成的（例如，一个规划者配置文件被分配了 `T0: "调查 Postgres 迁移"`），用您创建的内容摘要将其标记为完成：

```python
kanban_complete(
    summary="分解为 T1-T4：2 个研究通道并行，1 个基于其输出的综合，1 个基于建议的散文草稿",
    metadata={
        "task_graph": {
            "T1": {"assignee": "<profile-A>", "parents": []},
            "T2": {"assignee": "<profile-A>", "parents": []},
            "T3": {"assignee": "<profile-B>", "parents": ["T1", "T2"]},
            "T4": {"assignee": "<profile-C>", "parents": ["T3"]},
        },
    },
)
```

### 步骤 5 — 向用户报告

用平实的语言告诉他们您创建了什么，命名您使用的实际配置文件：

> 我已排队了 4 个任务：
> - **T1** (`<profile-A>`): 成本比较
> - **T2** (`<profile-A>`): 性能比较，与 T1 并行
> - **T3** (`<profile-B>`): 将 T1 + T2 综合成一份建议
> - **T4** (`<profile-C>`): 将 T3 转化为 CTO 备忘录
>
> 调度器现在将拾取 T1 和 T2。T3 在两者完成后开始。当 T4 完成时，您会收到网关通知。使用仪表板或 `hermes kanban tail <id>` 来跟踪进度。

## 常见模式

**扇出 + 扇入（研究 → 综合）：** N 个研究风格的卡片没有父任务，一个综合卡片将它们全部作为父任务。

**并行实施 + 验证：** 一个实施者卡片进行更改，而一个探索者/研究员卡片验证配置、文档或源代码映射。审查者卡片可以依赖于两者。不要仅仅因为用户在一句话中提到两者，就让实施者承担不相关的验证。

**带门控的管道：** `规划者 → 实施者 → 审查者`。每个阶段的 `parents=[previous_task]`。审查者阻止或完成；如果审查者阻止，操作者通过反馈解除阻止并重新生成。

**同配置文件队列：** N 个任务，全部分配给同一个配置文件，它们之间没有依赖关系。调度器序列化 — 该配置文件按优先级顺序处理它们，在自己的记忆中积累经验。

**人机协作：** 任何任务都可以 `kanban_block()` 以等待输入。调度器在 `/unblock` 后重新生成。评论线程携带完整的上下文。

## 常见陷阱

**虚构不存在的配置文件名。** 调度器在生成未知的被分配者时会静默失败——任务卡将永远停留在 `ready` 状态。请始终使用您在步骤 0 中发现的配置文件进行分配；如有疑问，请询问用户。

**将独立的并行路径捆绑到一张任务卡中。** 如果用户要求两个独立的结果，请创建两张任务卡。例如：“修复阻塞项并检查模型变体”不是一项修复任务；请为修复创建一张修复器/工程师任务卡，为变体检查创建一张探索器/研究员任务卡，然后可以选择性地将评审依赖于两者。

**因措辞而过度链接。** “最后检查 X”仍可能与实施并行，如果 X 是静态配置、文档或源代码发现。仅当检查依赖于实施结果时，才将其链接在实施之后。

**忘记依赖链接。** 如果任务图表示 `研究 -> 实施 -> 评审`，请不要将所有任务创建为独立的就绪卡。使用父链接，以便实施/评审在其输入存在之前无法运行。

**重新分配与新建任务。** 如果评审者以“需要修改”为由阻止，请从评审者的任务创建一个新任务——不要带着严肃的表情重新运行同一个任务。新任务分配给原始实施者的配置文件。

**链接的参数顺序。** `kanban_link(parent_id=..., child_id=...)`——父任务在前。弄混它们会使错误的任务降级为 `todo`。

**如果形状取决于中间发现，则不要预先创建整个图。** 如果 T3 的结构取决于 T1 和 T2 的发现，请让 T3 作为“综合发现”任务存在，其自身的第一步是读取父任务的交接成果并规划其余部分。协调者可以生成协调者。

**租户继承。** 如果您的环境变量中设置了 `HERMES_TENANT`，请在每次 `kanban_create` 调用时传递 `tenant=os.environ.get("HERMES_TENANT")`，以便子任务保持在同一命名空间内。

## 恢复卡住的工作进程

当某个工作进程配置文件持续崩溃、产生幻觉或被其自身的错误阻塞时（通常是：错误的模型、缺失的技能、损坏的凭据），看板仪表板会在任务上用 ⚠ 标记，并在抽屉中打开一个**恢复**部分。三个主要操作：

1.  **回收**（或 `hermes kanban reclaim <task_id>`）——立即中止正在运行的工作进程，并将任务重置为 `ready`。现有的声明 TTL 约为 15 分钟；这是快速脱困的途径。
2.  **重新分配**（或 `hermes kanban reassign <task_id> <new-profile> --reclaim`）——将任务切换到不同的配置文件（在此设置中存在的一个），并让调度器用一个全新的工作进程接起它。
3.  **更改配置文件模型**——仪表板会为 `hermes -p <profile> model` 打印一个可复制粘贴的提示，因为配置文件配置存储在磁盘上；在终端中编辑它，然后回收以使用新模型重试。

幻觉警告会出现在这样的任务上：工作进程的 `kanban_complete(created_cards=[...])` 声明包含了不存在或并非由该工作进程配置文件创建的任务卡 ID（门控会阻止完成），或者其中的自由格式摘要引用了无法解析的 `t_<hex>` ID（建议性文本扫描，非阻塞）。两者都会生成审计事件，即使在恢复操作后也会持续存在——调试记录会保留下来。
