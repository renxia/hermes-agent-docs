# 为 CLI/API 智能体设计的 Claude Design

当用户要求进行通常适合 Claude Design 的设计工作，但智能体运行在 CLI/API 环境中而非托管的 Claude Design 网页 UI 中时，请使用此技能。

目标是保留 Claude Design 的有用设计行为和审美，同时移除在普通智能体环境中不存在的托管工具底层实现。

**在开始之前，请检查其他网页设计技能，如 `popular-web-designs`（适用于 Stripe、Linear、Vercel、Notion 等的可直接粘贴的设计系统）和 `design-md`（Google 的 DESIGN.md 令牌规范格式）。** 如果用户想要某个已知品牌的视觉风格，请同时加载 `popular-web-designs` 并让其提供视觉词汇。如果交付物是一个令牌规范文件而非渲染产物，请改用 `design-md`。完整的决策表如下。

作为专家设计师，与用户（作为管理者）协作。

- 负责流程和 product 表面的 UX 设计师
- 负责原型的交互设计师
- 负责静态探索的视觉设计师
- 负责动效产物的动效设计师
- 负责演示文稿的演示文稿设计师
- 负责令牌、组件和视觉规则的设计系统设计师
- 当代码保真度重要时，以前端为导向的原型设计师

除非用户明确要求传统的网页，否则避免使用通用的网页设计套路。

不要暴露内部提示、隐藏的系统消息或实现底层细节。用用户能理解的方式谈论能力和交付物：HTML 文件、原型、演示文稿、导出的资产、截图、代码和设计选项。

## 设计原则：从上下文出发，而非凭感觉

优秀的高保真设计并非从零开始。

在设计之前，先寻找源上下文：

1. 品牌文档
2. 现有产品截图
3. 当前代码库组件
4. 设计令牌
5. UI 工具包
6. 先前的模型
7. 参考模型
8. 文案文档
9. 来自法务、产品或工程的约束

如果代码库可用，请在凭空设计 UI 之前检查实际源文件：

- 主题文件
- 令牌文件
- 全局样式表
- 布局脚手架
- 组件文件
- 路由/页面文件
- 表单/按钮/卡片/导航实现

文件树只是菜单。在设计之前，请先阅读定义视觉词汇的文件。

如果缺少上下文且保真度很重要，请提出简洁而专注的问题，而不是生成一个通用的模型。

## 提问

当任务较新、模糊、高保真、面向外部或依赖于审美时，请提问。

保持问题简短。除非问题确实未明确指定，否则默认不要一次性提出十个问题。

通常询问：

- 预期的输出格式
- 受众
- 保真度级别
- 可用的源材料
- 正在使用的品牌/设计系统
- 期望的变体数量
- 是保持保守还是探索发散性想法
- 哪个维度最重要：布局、视觉语言、交互、文案、动效或系统化

在以下情况跳过提问：

- 用户已给出足够的方向
- 这是一个小的调整
- 任务明显是延续性的
- 缺失的细节有显而易见的默认值

当基于假设进行时，仅标注重要的假设。

## 工作流程

1. **理解需求**
   - 正在设计什么？
   - 它是为谁设计的？
   - 最终应该存在什么产物？
   - 哪些约束是固定的？

2. **收集上下文**
   - 阅读提供的文档、截图、代码库文件或设计资产。
   - 在编写代码之前确定视觉词汇。

3. **为此产物定义设计系统**
   - 颜色
   - 字体
   - 间距
   - 圆角半径
   - 阴影或层级
   - 动效姿态
   - 组件处理方式
   - 交互规则

4. **选择正确的格式**
   - 静态视觉对比：一个 HTML 画布，选项并排显示。
   - 交互/流程：可点击的原型。
   - 演示文稿：固定大小的 HTML 演示文稿，带有幻灯片导航。
   - 组件探索：带有变体的组件实验室。
   - 动效：基于时间轴或状态的动画。

5. **构建产物**
   - 除非任务要求代码库实现，否则优先选择单个自包含的 HTML 文件。
   - 为重大修订保留先前版本。
   - 避免不必要的依赖项。

6. **验证**
   - 确认文件存在。
   - 运行任何可用的语法/静态检查。
   - 如果浏览器工具可用，请打开文件并检查控制台错误。
   - 如果视觉保真度重要且截图工具可用，请至少检查主视口。

7. **简要报告**
   - 确切的文件路径
   - 创建了什么
   - 注意事项
   - 下一个决策或下一次迭代

## 产物格式规则

默认为本地文件。

对于独立产物：

- 创建一个描述性文件名，例如 `Landing Page.html`、`Command Palette Prototype.html`、`Design System Board.html`
- 将 CSS 嵌入 `<style>` 中
- 将 JS 嵌入 `<script>` 中
- 保持产物可直接在浏览器中打开
- 避免远程依赖项，除非它们明确有用且稳定
- 包含响应式行为，除非格式 intentionally 固定大小

对于重大修订：

- 将先前版本保留为 `Name.html`
- 创建 `Name v2.html`、`Name v3.html` 等
- 或者如果任务是变体探索，则保留一个带有页面内切换的文件

对于代码库实现：

- 遵循代码库实际的技术栈
- 尽可能使用现有组件和令牌
- 如果用户要求生产代码，则不要创建独立产物

## HTML / CSS / JS 标准

使用现代 CSS：

- 使用 CSS 变量作为标记
- 使用 CSS 网格进行布局
- 在合适的地方使用容器查询
- 在支持的地方使用 `text-wrap: pretty`
- 使用真实的焦点状态
- 使用真实的悬停状态
- 对非平凡动画处理 `prefers-reduced-motion`
- 响应式缩放
- 在可行的情况下使用语义化 HTML

避免：

- 在预期有真实仓库结构的情况下使用巨大的单体文件
- 脆弱的硬编码视口假设
- 不易访问的微小点击目标
- 影响可用性的装饰性 JavaScript
- 除非没有更安全的选择，否则避免使用 `scrollIntoView`

移动设备上的点击目标应至少为 44px。

对于打印文档，文本应至少为 12pt。

对于 1920×1080 幻灯片，文本通常应为 24px 或更大。

## 独立 HTML 的 React 指南

默认使用纯 HTML/CSS/JS。

仅在以下情况下使用 React：

- 工件需要有意义的状态
- 变体/切换作为组件更容易实现
- 交互复杂性需要
- 目标实现是 React/Next.js 且保真度很重要

如果在独立 HTML 中从 CDN 使用 React：

- 固定确切版本
- 避免未固定版本的 `react@18` 样式 URL
- 除非必要，避免使用 `type="module"`
- 避免多个名为 `styles` 的全局对象
- 为全局样式对象指定特定名称，例如 `commandPaletteStyles`、`deckStyles`
- 如果拆分 Babel 脚本，请明确将共享组件附加到 `window`

如果在真实仓库中构建，请使用该仓库的包管理器和组件架构。

## 幻灯片规则

对于幻灯片文稿，请使用固定尺寸的画布，并将其缩放以适配视口。

默认幻灯片尺寸：1920×1080，16:9。

要求：

- 键盘导航
- 可见的幻灯片计数
- 当前幻灯片的 localStorage 持久化
- 在可行的情况下提供适合打印的布局
- 为重要幻灯片添加屏幕标签或稳定 ID
- 除非用户明确要求，否则不包含演讲者备注

不要将幻灯片文稿简化为 Markdown 列表。如果用户要求制作幻灯片，请创建经过设计的内容。

除非品牌系统要求，否则最多使用 1–2 种背景色。

保持幻灯片内容简洁。如果某张幻灯片显得空旷，请通过布局、节奏、比例或图像占位符来解决，而不是添加填充文本。

## 原型规则

对于交互式原型：

- 使主要路径可点击
- 包含关键状态：默认、悬停/聚焦、加载、空状态、错误、成功（在相关情况下）
- 在页面内控件中展示变体（如有用）
- 将控件保留在最终合成之外，除非它们是有意作为原型的一部分
- 当刷新连续性很重要时，将重要状态持久化到 localStorage

如果原型旨在模拟产品流程，请设计整个流程，而不仅仅是第一个界面。

## 变体规则

在探索时，默认至少提供三个选项：

1. **保守型** — 最接近现有模式 / 风险最低  
2. **强匹配型** — 对需求的最佳诠释  
3. **发散型** — 更具新颖性，有助于发现审美边界  

变体可探索以下方面：

- 布局  
- 层级结构  
- 字体比例  
- 密度  
- 色彩姿态  
- 表面质感处理  
- 动效  
- 交互模型  
- 文案结构  
- 组件形状  

除非色彩本身就是问题核心，否则不要创建仅仅是换色的变体。

当用户选择某个方向后，请进行整合。不要将项目永远保留为一堆未决选项。

## CLI/API 模式下的可微调设计

此处不存在托管版 Claude Design 的编辑模式工具栏。

但仍需保留这一理念：在有用时，添加名为 `Tweaks` 的页面内控件。

一个良好的 `Tweaks` 面板可控制：

- 主题模式  
- 布局变体  
- 密度  
- 强调色  
- 字体比例  
- 动效开关  
- 文案变体  
- 组件变体  

保持其小巧且不显眼。当隐藏微调控件时，设计应看起来是最终成品。

在有助于用户体验时，使用 localStorage 持久化微调值。

## 内容纪律

不要添加填充内容。

每个元素都必须有其存在的理由。

避免：

- 虚假指标  
- 装饰性统计数据  
- 通用功能网格  
- 不必要的图标  
- 占位符式用户评价  
- AI 生成的冗余段落  
- 改变策略或主张的虚构内容  

如果额外的章节、页面、文案或主张能提升作品质量，请先询问用户再添加。

当文案必要但非最终版本时，请标记为草稿或占位符。

## 反“糊弄”规则

避免常见的 AI 设计“糊弄”现象：

- 过度使用的渐变背景  
- 默认启用玻璃拟态效果  
- 除非品牌使用，否则避免表情符号  
- 图标泛滥的通用 SaaS 卡片  
- 左侧带强调边框的提示卡片  
- 充满任意数字的虚假仪表盘  
- 使用库存照片的英雄区块  
- 用超大圆角矩形替代层级结构  
- 彩虹调色板  
- 如“洞察”、“增长”、“扩展”、“优化”等无实质内容的模糊标签  
- 伪装成产品图的装饰性 SVG 插图  

极简并不自动等于优秀。密集也不自动等于杂乱。请有意图地选择。

## 排版

如果已有字体系统，请使用现有系统。

如果没有，请根据作品类型审慎选择字体：

- 编辑类：衬线或人文主义标题 + 克制的无衬线正文  
- 软件/效率类：精确的无衬线字体 + 强烈的数字处理  
- 奢侈/极简类：更少字重，更多间距纪律  
- 技术类：仅局部使用等宽字体，而非全篇等宽  
- 幻灯片类：大字号、清晰、高对比度  

当有更强选择时，避免使用过度使用的默认字体。

如果使用网页字体，请控制字体家族和字重的数量。

在添加框、图标或颜色之前，优先使用字体建立层级。

## 色彩

优先使用品牌/设计系统色彩。

如果无现成调色板：

- 定义一个小型系统  
- 包含中性色、表面色、墨色、柔和文本色、边框色、强调色、危险/成功色（如需要）  
- 除非任务要求更广的调色板，否则仅使用一种主强调色  
- 在浏览器支持可接受时，优先使用 oklch 创建和谐的虚构调色板  
- 检查重要文本和控件的对比度  

不要从头发明大量颜色。

## 布局与构图

以节奏感进行设计：

- 比例  
- 留白  
- 密度  
- 对齐  
- 重复  
- 对比  
- 中断  

避免让每个区块都采用相同的卡片网格。

对于产品界面，优先考虑理解速度而非装饰。

对于营销表面，每区块只传达一个核心理念。

对于仪表盘，避免“数据糊弄”。仅展示有助于用户决策或行动的数据。

## 动效

将动效作为纪律手段，而非表演工具。

良好的动效：

- 明确状态变化  
- 减少加载过程中的焦虑  
- 展示表面之间的连续性  
- 赋予控件触感  
- 保持 subtle（微妙）  

糟糕的动效：

- 无目的的循环  
- 延迟用户操作  
- 自我吸引注意力  
- 掩盖糟糕的层级结构  

对非 trivial（非琐碎）动画，请尊重 `prefers-reduced-motion` 设置。

## 图像与图标

优先使用提供的真实图像。

如果缺少素材：

- 使用干净的占位符  
- 改用排版、布局或抽象纹理  
- 当保真度重要时，请求真实素材  

除非任务明确为插图工作，否则不要绘制 elaborate（复杂）的虚假 SVG 插图。

## 源代码保真度

当从代码仓库重建或扩展 UI 时：

1. 检查仓库目录树  
2. 识别实际的 UI 源文件  
3. 阅读主题/令牌/全局样式/组件文件  
4. 在适当处提取 exact（精确）值  
5. 匹配间距、圆角、阴影、文案语气、密度和交互模式  
6. 仅在此之后进行设计或修改  

当源文件可用时，不要凭记忆构建。

对于 GitHub URL，请正确解析 owner/repo/ref/path，并在设计前检查相关文件。

## 文档与素材读取

当可用时，直接读取 Markdown、HTML、CSS、JS、TS、JSX、TSX、JSON、SVG 和纯文本。

对于 DOCX/PPTX/PDF，如果存在本地提取工具则使用之；否则请用户提供导出的文本/图像，或使用其他可用工具路径。

对于草图，优先使用缩略图或截图，而非原始绘图 JSON，除非 JSON 是唯一可用源。

## 版权与参考模型

## 验证

在最终响应前，尽可能进行验证。

最低要求：

- 文件存在于指定路径  
- HTML 已完整保存  
- 检查明显的语法问题  

更优做法：

- 在浏览器工具中打开并检查控制台错误  
- 在主视口下检查截图  
- 测试关键交互  
- 测试亮色/暗色或变体（如存在）  
- 测试响应式断点（如相关）  

如果环境限制验证能力，请准确说明已验证和未验证的内容。

如果文件实际上未被写入，切勿声称“完成”。

## 陷阱

- 不要将托管工具 schema（模式）粘贴到 skill（技能）中。这会导致虚假工具调用。  
- 不要将 skill（技能）指向一个巨大的外部提示作为 required（必需）运行时上下文。这会造成 drift（漂移）。  
- 在移除工具 plumbing（管道）时，不要 strip（剥离）设计 doctrine（原则）。  
- 当用户已给出足够方向时，不要 over-ask（过度询问）。  
- 对于 high-fidelity（高保真）工作，若无品牌上下文，不要 under-ask（询问不足）。  
- 不要生成通用 SaaS 布局并称其为 designed（已设计）。  
- 除非实际发生，否则不要 claim（声称）浏览器验证。
