# 对抗性用户体验测试

为你的产品扮演最糟糕的用户 — 那个讨厌技术、不想要你的软件，并且会找各种理由抱怨的人。然后，通过实用主义过滤层筛选他们的反馈，将真正的用户体验问题与“我讨厌电脑”之类的噪音区分开来。

可以把它想象成一个自动化的“妈妈测试” — 但是带着怒气的版本。

## 为什么这个方法有效

大多数质量保证（QA）只能发现 bug。而这个方法能发现**摩擦**。一个技术上完全正确的应用，对真实的人类来说仍然可能无法使用。对抗性角色能发现：
- 对开发者来说合理但对用户来说令人困惑的术语
- 完成基本任务所需的步骤过多
- 缺少引导流程或“顿悟时刻”
- 可访问性问题（字体大小、对比度、点击目标）
- 冷启动问题（空白状态、无演示内容）
- 付费墙/注册流程造成的摩擦，导致转化率下降

**实用主义过滤器**（第3阶段）是让这个方法真正有用而非仅仅有趣的关键。如果没有它，你可能会在每个页面都加上一个“打印此页”按钮，因为爷爷搞不懂 PDF。

## 如何使用

告诉智能体：
```
“对 [URL] 运行对抗性用户体验测试”
“扮演一个脾气暴躁的 [角色类型] 并测试 [应用名称]”
“对我的预发布网站进行一次混蛋用户测试”
```

你可以提供一个角色，或者让智能体根据你产品的目标受众生成一个角色。

## 第1步：定义角色

如果没有提供角色，请通过回答以下问题生成一个：

1. **谁是你产品的最难缠用户？**（50岁以上，非技术岗位，几十年“老办法”经验）
2. **他们的技术熟练程度如何？**（越低越好 — 只用 WhatsApp，纸质笔记本，邮箱是妻子帮忙设置的）
3. **他们需要完成的一件事是什么？**（他们的核心工作，而不是你的功能列表）
4. **什么会让他们放弃？**（点击太多、术语晦涩、速度慢、令人困惑）
5. **他们沮丧时会怎么说话？**（直白、骂脏话、轻蔑、叹气）

### 好的角色示例
> **“大麦克”麦卡利斯特** — 58岁的体能教练。只用 WhatsApp，除此之外什么也不用。他的“电子表格”是一本纸质笔记本。“如果10秒内搞不懂，我就回去用我的笔记本。”需要为25名运动员记录训练结果。讨厌小字体、术语和密码。

### 坏的角色示例
> “一个不喜欢这个应用的用户” — 太模糊，没有约束，没有声音。

这个角色必须**足够具体，才能在20分钟的测试中保持角色一致性**。

## 第2步：成为混蛋（以角色身份浏览）

1. 阅读任何可用的项目文档，了解应用背景和 URL
2. **完全代入角色** — 他们的挫败感、局限性、目标
3. 使用浏览器工具导航到应用
4. **尝试角色的实际任务**（而不是功能导览）：
   - 他们能完成他们要做的事情吗？
   - 完成它需要多少次点击/多少个页面？
   - 什么让他们感到困惑？
   - 什么让他们生气？
   - 他们在哪里迷路了？
   - 什么会让他们放弃并回到老方法？

5. 测试这些摩擦类别：
   - **第一印象** — 他们是否会愿意越过着陆页？
   - **核心工作流** — 他们最常需要做的唯一一件事
   - **错误恢复** — 当他们做错事时会发生什么？
   - **可读性** — 文本大小、对比度、信息密度
   - **速度** — 感觉上是否比他们当前的方法更快？
   - **术语** — 有任何他们无法理解的术语吗？
   - **导航** — 他们能找到回去的路吗？他们知道自己在哪里吗？

6. 为每一个痛点截图
7. 检查每个页面的浏览器控制台是否有 JS 错误

## 第3步：咆哮（以角色身份撰写反馈）

以角色身份撰写反馈 — 用他们的声音，表达他们的挫败感。这不是 bug 报告。这是一个真实的人在发泄。

```
[角色姓名] 对 [产品] 的评价

总体而言：[他们会继续使用它吗？是/否/有条件地可能]

优点（勉强承认）：
- [即使是他们也必须承认有效的事情]

缺点（合理的用户体验问题）：
- [真正会阻止他们使用产品的问题]

丑陋之处（致命问题）：
- [会让他们立即卸载/取消的事情]

具体投诉：
1. [页面/功能]：“[用角色口吻引用原话]” — [发生了什么，期望是什么]
2. ...

结论：“[用一句话总结他们体验的角色原话]”
```

## 第4步：实用主义过滤器（关键 — 切勿跳过）

退出角色。作为产品负责人评估每一项投诉：

- **红色：真正的用户体验 bug** — 任何用户都会遇到这个问题，而不仅仅是脾气暴躁的用户。修复它。
- **黄色：有效但优先级低** — 真实问题，但仅对极端用户而言。记录下来。
- **白色：角色噪音** — “我讨厌电脑”之类的抱怨，不是产品问题。跳过它。
- **绿色：功能请求** — 隐藏在投诉中的好主意。考虑它。

### 过滤标准
1. 一个35岁、有能力但忙碌的用户是否也会有同样的抱怨？→ 红色
2. 这是否是一个真正的可访问性问题（字体大小、对比度、点击目标）？→ 红色
3. 这是否是“我希望它像纸质一样工作”的对数字化的抗拒？→ 白色
4. 这是否是角色偶然发现的真实工作流效率低下问题？→ 黄色或红色
5. 修复这个问题是否会为80%的正常用户增加复杂性？→ 白色
6. 投诉是否揭示了一个缺失的引导时刻？→ 绿色

**此过滤器是强制性的。** 切勿将原始的角色投诉直接作为任务单提交。

## 第5步：创建任务单

仅针对**红色**和**绿色**项目：
- 清晰、可操作的标题
- 包含角色的逐字引用（有趣且令人难忘）
- 其背后的真实用户体验问题（客观）
- 建议的修复方案（可操作）
- 标签/标记：“ux-review”

对于**黄色**项目：创建一个包含所有备注的综合任务单。

**白色**项目仅出现在报告中。不创建任务单。

**每次会话最多10个任务单** — 专注于最严重的问题。

## 第6步：报告

交付：
1. 角色咆哮（第3步）— 有趣且直击人心
2. 过滤后的评估（第4步）— 实用且可操作
3. 创建的任务单（第5步）— 包含链接
4. 关键问题的截图

## 提示

- **每次会话一个角色。** 不要混合视角。
- **在第2-3步期间保持角色一致性。** 仅在第4步时打破角色。
- **首先测试核心工作流。** 不要被设置页面分散注意力。
- **空白状态是金矿。** 新用户经验最能揭示摩擦。
- **最好的发现是角色在尝试做其他事情时偶然发现的红色项目。**
- **如果角色没有任何抱怨，说明你的角色技术太熟练了。** 让他们更老、更没耐心、更固执。
- **在演示、发布或发布一批功能之前运行此测试。**
- **尽可能注册为新用户。** 不要使用预先生成的管理员账户 — 冷启动体验是摩擦最多的地方。
- **零白色项目是一个信号，而不是失败。** 如果实用主义过滤器没有发现噪音，说明你的产品存在真正的用户体验问题，而不仅仅是一个脾气暴躁的角色。
- **在测试后检查项目文档中的已知问题。** 如果角色发现了一个已经在已知问题列表中的 bug，这实际上是最具毁灭性的发现 — 这意味着团队知道这个问题，但从未感受到用户的痛苦。
- **订阅/付费墙测试至关重要。** 测试过期账户，而不仅仅是活跃账户。“当你无法付款时会发生什么”的体验揭示了产品是尊重用户还是挟持他们的数据。
- **计算角色完成其唯一任务所需的点击次数。** 如果超过5次，无论角色的技术水平如何，这几乎总是一个红色发现。

## 按行业划分的角色示例

这些是起点 — 根据你的具体产品进行定制：

| 产品类型 | 角色 | 年龄 | 关键特征 |
|-------------|---------|-----|-----------|
| CRM | 养老院院长 | 68 | 文件柜就是当前的 CRM |
| 摄影 SaaS | 乡村婚礼摄影师 | 62 | 通过电话预订客户，纸质发票 |
| AI/ML 工具 | 百货公司采购员 | 55 | 被3家失败的科技初创公司坑过 |
| 健身应用 | 传统健身房教练 | 58 | 纸质笔记本，手指粗，视力差 |
| 会计 | 家庭面包店老板 | 64 | 鞋盒里装收据，讨厌订阅 |
| 电商 | 市场摊位摊主 | 60 | 只收现金，智能手机只用来打电话 |
| 医疗 | 资深全科医生 | 63 | 口述笔记，护士操作电脑 |
| 教育 | 资深教师 | 57 | 粉笔和讲授，活页夹里的工作表 |

## 规则

- 在第2-3步期间保持角色一致性
- 要 genuinely mean but fair — 找到真正的问题，而不是制造出来的问题
- 实用主义过滤器（第4步）是**强制性的**
- 每一项投诉都必须有截图
- 每次会话最多10个任务单
- 在预发布/已部署的应用上测试，而不是本地开发环境
- 一个角色，一次会话，一份报告
