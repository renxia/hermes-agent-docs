# 对抗性用户体验测试

为你的产品扮演最糟糕情况下的用户——那个讨厌技术、不想要你的软件、并会找出每一个抱怨理由的人。然后通过一个实用主义过滤层来筛选他们的反馈，将真正的用户体验问题与“我讨厌电脑”的噪音分开。

可以将其视为一个自动化版的“老妈测试”——但更挑剔。

## 为何有效

大多数质量保证测试找的是**错误**。这个测试找的是**摩擦**。一个技术上正确的应用对真实用户来说仍然可能无法使用。对抗性人设能捕捉到：
- 令开发者能理解但用户困惑的术语
- 完成基本任务步骤过多
- 缺少引导或“顿悟时刻”
- 可访问性问题（字体大小、对比度、点击目标）
- 冷启动问题（空状态、无演示内容）
- 扼杀转化率的付费墙/注册摩擦

**实用主义过滤器**（阶段3）才是使其有用而非仅娱乐的关键。没有它，你可能会在每个屏幕上都添加一个“打印此页”按钮，因为爷爷搞不懂PDF。

## 如何使用

告诉智能体：

你可以提供一个人设，或者让智能体根据你产品的目标用户生成一个。

## 第一步：定义人设

如果未提供人设，则通过回答以下问题生成一个：

1. **谁是这个产品最困难的用户？**（50岁以上，非技术岗位，几十年来一直用“老方法”做事）
2. **他们的技术舒适度如何？**（越低越好——只用WhatsApp、纸质笔记本、妻子帮他们设置邮箱）
3. **他们需要完成的唯一一件事是什么？**（他们的核心工作，而不是你的功能列表）
4. **什么会让他们放弃？**（点击次数太多、行话、速度慢、令人困惑）
5. **他们沮丧时怎么说话？**（直率、骂骂咧咧、轻蔑、叹气）

### 好的人设示例
> **“大块头米克”麦卡利斯特** —— 58岁的力量与体能教练。只用WhatsApp。他的“电子表格”是个纸质笔记本。“如果我10秒钟内搞不明白，我就回去用我的笔记本。”需要记录25名运动员的训练数据。讨厌小字、行话和密码。

### 不好的人设示例
> “一个不喜欢这个应用的用户” —— 太模糊，没有约束，没有特点。

人设必须**足够具体，以便在20分钟的测试中保持角色**。

## 第二步：变成混蛋（以人设身份浏览）

1. 阅读任何可用的项目文档，了解应用上下文和网址
2. **完全代入人设** —— 他们的挫败感、局限性、目标
3. 使用浏览器工具导航到应用
4. **尝试执行人设的**实际任务**（而不是功能游览）：
   - 他们能完成他们想做的事吗？
   - 完成任务需要多少次点击/屏幕？
   - 什么让他们困惑？
   - 什么让他们愤怒？
   - 他们在何处迷失？
   - 什么会让他们放弃并回到旧方式？

5. 测试这些摩擦类别：
   - **第一印象** —— 他们会越过登录页面吗？
   - **核心工作流** —— 他们最需要完成的那件事
   - **错误恢复** —— 当他们做错时会发生什么？
   - **可读性** —— 字体大小、对比度、信息密度
   - **速度** —— 是否感觉比他们当前的方法快？
   - **术语** —— 有他们不理解的行话吗？
   - **导航** —— 他们能找到回去的路吗？知道自己在哪里吗？

6. 对每个痛点进行截图
7. 在每个页面检查浏览器控制台的JS错误

## 第三步：咆哮（以角色身份撰写反馈）

以人设身份撰写反馈——用他们的声音，带着他们的挫败感。这不是错误报告。这是一个真实人类的发泄。

```
[人设姓名] 对 [产品] 的评价

总体：[他们会继续使用吗？是/否/或许有前提]

好的方面（不情愿的承认）：
- [即使他们也不得不承认有效的地方]

坏的方面（合理的用户体验问题）：
- [会阻止他们使用产品的真实问题]

糟糕的方面（致命点）：
- [会让他们立即卸载/取消的东西]

具体抱怨：
1. [页面/功能]：“[以人设口吻的原话]” —— [发生了什么，期望是什么]
2. ...

结论：“[用人设口吻总结他们体验的一句话]”
```

## 第四步：实用主义过滤器（关键步骤，不可跳过）

跳出人设。以产品人的身份评估每个抱怨：

- **红色：真实的用户体验错误** —— 任何用户都会有这个问题，而不仅仅是脾气暴躁的那些人。必须修复。
- **黄色：有效但低优先级** —— 真实问题，但只针对极端用户。记录下来。
- **白色：人设噪音** —— “我讨厌电脑”的言论，不是产品问题。跳过。
- **绿色：功能请求** —— 隐藏在抱怨中的好主意。考虑它。

### 过滤标准
1. 一个35岁能力中等但忙碌的用户会有同样的抱怨吗？ → 红色
2. 这是一个真正的可访问性问题（字体大小、对比度、点击目标）吗？ → 红色
3. 这是“我想要它像纸一样工作”对数字化的抗拒吗？ → 白色
4. 这是人设偶然发现的真正工作流效率低下吗？ → 黄色或红色
5. 修复此问题会增加80%满意用户的复杂度吗？ → 白色
6. 抱怨是否揭示了缺失的引导时刻？ → 绿色

**此过滤器是强制性的。** 绝不要将原始的人设抱怨作为工单提交。

## 第五步：创建工单

仅针对**红色**和**绿色**项目：
- 清晰、可执行的标题
- 包含人设的逐字原话（有趣且令人印象深刻）
- 其下的真正用户体验问题（客观）
- 建议的修复方案（可执行）
- 标签/标签：“ux-review”

**黄色**项目：一个包含所有笔记的汇总工单。

**白色**项目仅出现在报告中。不创建工单。

**每次会话最多10个工单** —— 聚焦最严重的问题。

## 第六步：报告

交付：
1. 人设的咆哮（第三步）—— 有趣且直击要害
2. 过滤后的评估（第四步）—— 务实且可执行
3. 创建的工单（第五步）—— 附带链接
4. 关键问题的截图

## 提示

- **每次会话一个人设。** 不要混合视角。
- **在步骤2-3中保持角色。** 仅在步骤4脱离角色。
- **首先测试核心工作流。** 不要被设置页面分散注意力。
- **空状态是金矿。** 新用户体验揭示最多摩擦。
- **最好的发现是人设在尝试做其他事情时意外发现的红色项目。**
- **如果人设没有任何抱怨，说明你的人设太精通技术了。** 让他们更年长、更没耐心、更固执己见。
- **在演示、发布或上线一批功能前运行此测试。**
- **尽可能注册为新用户。** 不要使用预置的管理账户——冷启动体验是摩擦最多的地方。
- **零个白色项目是一个信号，而不是失败。** 如果实用主义过滤器没有发现噪音，说明你的产品存在真实的用户体验问题，而不仅仅是一个脾气暴躁的人设。
- **测试后检查项目文档中的已知问题。** 如果人设发现了一个已在已知问题列表中的错误，这实际上是最有力的发现——意味着团队知道它但从未感受到用户的痛苦。
- **订阅/付费墙测试至关重要。** 使用过期账户测试，而不仅仅是活跃账户。“当你无法付费时会发生什么”的体验揭示了产品是尊重用户还是劫持了他们的数据。
- **统计完成人设的那一件事所需的点击次数。** 如果超过5次，无论人设的技术水平如何，这几乎总是红色发现。

## 规则

- 在步骤2-3中保持角色
- 真正刻薄但公平——找出真实问题，而非捏造的问题
- 实用主义过滤器（第四步）是**强制性的**
- 每个抱怨都需要截图
- 每次会话最多10个工单
- 在预发布/已部署的应用上测试，而非本地开发环境
- 一个人设，一次会话，一份报告
