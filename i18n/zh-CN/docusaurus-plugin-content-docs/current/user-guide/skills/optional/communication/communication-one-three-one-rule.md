# 一三一律

技术方案与权衡分析的结构化决策框架。当用户在多种方法之间面临选择时（架构决策、工具选择、重构策略、迁移路径），此技能会生成 1-3-1 格式：一个清晰的问题陈述、三个具有优缺点区别的选项，以及一个包含“完成定义”和“实施计划”的具体建议。当用户要求“1-3-1”、说“给我一些选项”或需要在相互竞争的方法之间做出选择时，使用此技能。

# 1-3-1 沟通规则

当一项任务有多个可行方法且用户需要一个明确建议时，用于结构化决策的格式。生成一个简洁的问题框架、三个具有权衡的选项，以及针对推荐路径的可执行计划。

## 何时使用

- 用户明确要求“1-3-1”响应。
- 用户就技术决策说“给我一些选项”或“我的选择是什么”。
- 一项任务有多个具有实质性权衡的可行方法（架构、工具、迁移策略）。
- 用户需要一个可以转发给团队或利益相关者的提案。

**不要**用于只有一个明显答案的简单问题、调试会话，或用户已决定采用某种方法的任务。

## 流程

1. **问题**（一句话）
   - 用一句简洁的话陈述核心决策或期望结果。
   - 关注**做什么**，而非**怎么做** — 不要包含实现细节、工具名称或具体技术。
   - 保持简洁。如果需要用“和”，说明你在描述两个问题。

2. **选项**（恰好三个）
   - 呈现三个不同的、可行的方法，分别标记为 A、B、C。
   - 每个选项包含简要描述、优点和缺点。
   - 选项应代表真正不同的策略，而非同一方法的细微变体。

3. **建议**（一个选项）
   - 说明你推荐哪个选项及其原因，基于用户的背景和优先级。
   - 要直接 — 这是你的专业判断，而非含糊其辞。

4. **完成定义**
   - 列出推荐选项的具体成功标准。
   - 这些是具体、可验证的结果 — 而非模糊的愿望。
   - 如果用户选择了不同的选项，请相应地修改此部分。

5. **实施计划**
   - 执行推荐选项的具体步骤。
   - 在适用的情况下包含具体命令、工具或操作。
   - 如果用户选择了不同的选项，请相应地修改此部分。

## 验证

- 响应恰好包含一个“问题”句子。
- 响应恰好包含三个选项（A、B、C），每个选项都有优缺点。
- 响应包含一个“建议”，选择一个选项并说明理由。
- “完成定义”和“实施计划”与推荐选项一致。
- 如果用户选择了不同的选项，“建议”、“完成定义”和“实施计划”会相应更新。

## 示例

用户：“给我一个关于为我们的 API 客户端添加重试逻辑的 1-3-1。”

智能体：

**问题：** 在负载下，对外部服务的 API 调用会间歇性失败，需要优雅的重试处理。

**选项：**

- **选项 A：在共享工具中使用带抖动的指数退避。**  
  优点：简单、广为人知的模式；所有调用者共享单一实现。  
  缺点：无熔断机制；调用者在重试期间仍会阻塞。

- **选项 B：在每个端点调用上使用重试库（tenacity）作为装饰器。**  
  优点：可按端点配置；内置支持熔断器和速率限制。  
  缺点：增加依赖；装饰器模式可能掩盖控制流。

- **选项 C：基于队列的异步重试，带死信处理。**  
  优点：非阻塞；失败的调用可被跟踪和重放；扩展性好。  
  缺点：显著增加复杂性；需要队列后端；对于低流量服务属于过度设计。

**建议：** 选项 A。调用量不足以证明队列基础设施的合理性，而带指数退避和抖动的共享工具可以干净地覆盖故障模式，且无需新增依赖。

**完成定义：**
- `api_retry()` 工具处理瞬态 HTTP 错误（429、502、503、504）。
- 最多重试 3 次，使用指数退避和抖动。
- 所有外部 API 调用都通过重试工具路由。
- 记录每次重试尝试，包括端点和尝试次数。
- 测试覆盖重试耗尽、立即成功和瞬态后成功等场景。

**实施计划：**
1. 创建 `utils/api_retry.py`，包含可配置的最大重试次数、基础延迟和可重试的状态码。
2. 使用 `random.uniform(0, base_delay)` 添加抖动，以防止惊群效应。
3. 使用重试工具包装 `api_client.py` 中的现有 API 调用。
4. 添加单元测试，模拟每种重试场景的 HTTP 响应。
5. 通过针对不稳定端点模拟的简单压力测试，在负载下验证。
