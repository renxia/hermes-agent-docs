# 持久目标（`/goal`）

`/goal` 为 Hermes 设定一个跨轮次保持有效的目标。每一轮结束后，一个轻量级评判模型会检查该目标是否已被智能体的最后一次响应满足。如果尚未满足，Hermes 会自动将一个延续提示反馈回同一会话并继续工作——直到目标达成、你暂停或清除它，或者轮次预算耗尽。

这是我们对于 **Ralph 循环**的实现，直接受到 Eric Traut（OpenAI）在 [Codex CLI 0.128.0 的 `/goal`](https://github.com/openai/codex) 启发。核心理念——让目标在多个轮次中保持有效，并在达成前不停止——源自他们。此处的实现是独立的，并适配了 Hermes 的架构。

## 何时使用

当你希望 Hermes 自主迭代而无需你在每一轮都重新提示时，请使用 `/goal`：

- “修复 `src/` 中的所有 lint 错误，并确保 `ruff check` 通过”
- “将功能 X 从仓库 Y 移植过来，包括测试，并使 CI 变为绿色”
- “调查为什么会话 ID 在运行中途压缩时有时会漂移，并撰写一份报告”
- “构建一个小型 CLI 工具，根据 EXIF 日期重命名文件，然后在 photos/ 文件夹中对其进行测试”

智能体仅执行一轮就停止的任务不需要 `/goal`。*否则你将不得不连续说三次“继续”*的任务，正是此功能大放异彩之处。

## 快速开始

```
/goal 修复 tests/hermes_cli/ 中所有失败的测试，并确保 scripts/run_tests.sh 对该目录通过
```

你将看到：

1. **目标已接受** — `⊙ 目标已设定（20 轮预算）：<你的目标>`
2. **第 1 轮运行** — Hermes 开始工作，就像你发送了一条普通消息作为目标一样。
3. **评判运行** — 在该轮结束后，评判模型决定 `done` 或 `continue`。
4. **必要时触发循环** — 如果为 `continue`，你将看到 `↻ 正在向目标继续（1/20）：<评判的理由>`，Hermes 会自动采取下一步。
5. **终止** — 最终你会看到 `✓ 目标已达成：<理由>` 或 `⏸ 目标已暂停 — 已使用 N/20 轮`。

## 命令

| 命令 | 功能说明 |
|---|---|
| `/goal <文本>` | 设定（或替换）长期目标。立即启动第一轮，因此你无需发送单独的消息。 |
| `/goal` 或 `/goal status` | 显示当前目标、其状态以及已使用的轮次。 |
| `/goal pause` | 停止自动延续循环，但不清除目标。 |
| `/goal resume` | 恢复循环（将轮次计数器重置为零）。 |
| `/goal clear` | 完全放弃该目标。 |

在 CLI 和所有网关平台（Telegram、Discord、Slack、Matrix、Signal、WhatsApp、SMS、iMessage、Webhook、API 服务器以及 Web 控制台）上的行为完全一致。

## 行为细节

### 评判模型

每一轮结束后，Hermes 会调用一个辅助模型，传入：

- 长期目标的文本
- 智能体最近一次最终响应（最后约 4 KB 的文本）
- 一个系统提示，指示评判模型以严格的 JSON 格式回复：`{"done": <布尔值>, "reason": "<一句话的理由>"}`

评判模型 deliberately 保守：仅当响应**明确**确认目标已完成、最终交付物已清晰生成，或目标无法实现/受阻时（视为 DONE 并附带受阻原因，以免在不可能完成的任务上浪费预算），它才会将目标标记为 `done`。

### 失败开放语义

如果评判模型出错（网络波动、响应格式错误、辅助客户端不可用），Hermes 会将裁决视为 `continue` —— 损坏的评判模型永远不会阻碍进度。**轮次预算**才是真正的最后保障。

### 轮次预算

默认值为 20 轮延续（`config.yaml` 中的 `goals.max_turns`）。当预算耗尽时，Hermes 会自动暂停并明确告知你如何继续：

```
⏸ 目标已暂停 — 已使用 20/20 轮。使用 /goal resume 继续，或使用 /goal clear 停止。
```

`/goal resume` 会将计数器重置为零，因此你可以分阶段继续。

### 用户消息始终优先

当目标处于活动状态时，你发送的任何真实消息都优先于延续循环。在 CLI 上，你的消息会进入 `_pending_input`，排在排队等待的延续之前；在网关上，它通过适配器 FIFO 以相同方式传递。你的轮次结束后，评判模型会再次运行 —— 因此如果你的消息恰好完成了目标，评判模型会捕获到并停止。

### 运行中途安全性（网关）

当智能体已在运行时，`/goal status`、`/goal pause` 和 `/goal clear` 是安全的 —— 它们仅触及控制平面状态，不会中断当前轮次。在运行中途设定**新**目标（`/goal <新文本>`）会被拒绝，并提示你首先使用 `/stop`，以防止旧的延续与新的延续竞争。

### 持久化

目标状态存储在 `SessionDB.state_meta` 中，键为 `goal:<会话 ID>`。这意味着 `/resume` 会从你离开的地方继续 —— 设定一个目标，合上笔记本电脑，明天回来，`/resume`，目标仍然保持你离开时的状态（活动、暂停或已完成）。

### 提示缓存

延续提示是一个附加到历史的普通用户角色消息。它**不会**修改系统提示、交换工具集，或以任何方式使 Hermes 的提示缓存失效。运行一个 20 轮的目标在缓存方面的成本与 20 轮正常对话相同。

## 配置

添加到 `~/.hermes/config.yaml`：

```yaml
goals:
  # Hermes 自动暂停并询问你 /goal resume 前的最大延续轮次。
  # 默认为 20。如果你希望循环更紧凑，请降低此值；
  # 对于长时间的重构，请提高此值。
  max_turns: 20
```

### 选择评判模型

评判使用 `goal_judge` 辅助任务。默认情况下，它解析为你的主模型（参见[辅助模型](/docs/user-guide/configuration#auxiliary-models)）。如果你希望将评判路由到一个廉价快速的模型以降低成本，请添加一个覆盖：

```yaml
auxiliary:
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

评判调用很小（约 200 个输出 token），每轮运行一次，因此廉价快速的模型通常是正确的选择。

## 示例演练

```
你: /goal 每轮创建一个文件 /tmp/note_{1..4}.txt，共四个文件，每个文件包含其编号作为文本

  ⊙ 目标已设定（20 轮预算）：每轮创建一个文件 /tmp/note_{1..4}.txt，共四个文件，每个文件包含其编号作为文本

Hermes: 正在创建 /tmp/note_1.txt。
  💻 echo "1" > /tmp/note_1.txt   (0.1s)
  我已创建了 /tmp/note_1.txt，内容为 "1"。根据你的要求，我将在下一轮继续处理剩余的文件。

  ↻ 正在向目标继续（1/20）：仅创建了 1 个文件（共 4 个）；还剩 3 个文件。

Hermes: [正在向你的长期目标继续]
  💻 echo "2" > /tmp/note_2.txt   (0.1s)
  已创建 /tmp/note_2.txt。还剩两个。

  ↻ 正在向目标继续（2/20）：已创建 2 个文件（共 4 个）；还剩 2 个。

Hermes: [正在向你的长期目标继续]
  💻 echo "3" > /tmp/note_3.txt   (0.1s)
  已创建 /tmp/note_3.txt。

  ↻ 正在向目标继续（3/20）：已创建 3 个文件（共 4 个）；还剩 1 个。

Hermes: [正在向你的长期目标继续]
  💻 echo "4" > /tmp/note_4.txt   (0.1s)
  所有四个文件均已创建：/tmp/note_1.txt 到 /tmp/note_4.txt，每个文件包含其编号。

  ✓ 目标已达成：所有四个文件均已创建，并包含指定内容，完成了目标。

你: _
```

四轮，一次 `/goal` 调用，你无需说一次“继续”。

## 当评判模型出错时

没有评判模型是完美的。需要注意两种失败模式：

**假阴性 —— 目标实际已完成，但评判模型说继续。** 轮次预算会捕获此情况。你会看到 `⏸ 目标已暂停`，可以 `/goal clear` 或仅发送一条新消息。

**假阳性 —— 工作仍有剩余，但评判模型说已完成。** 你会看到 `✓ 目标已达成`，但你知道并非如此。发送一条后续消息以继续，或更精确地重新设定目标：`/goal <更精确的文本>`。评判模型的系统提示 deliberately 保守，以使假阳性比假阴性更少见。

如果你发现评判模型的裁决缺乏说服力，`↻ 正在向目标继续` 或 `✓ 目标已达成` 行中的理由文本会明确告诉你评判模型看到了什么。这通常足以诊断是目标文本模糊，还是模型的响应存在问题。

## 归属

`/goal` 是 Hermes 对 **Ralph 循环**模式的实现。面向用户的设计 —— 让目标在多个轮次中保持有效，在达成前不停止，并提供创建/暂停/恢复/清除控制 —— 由 Eric Traut 在 OpenAI 的 Codex 团队于 [Codex CLI 0.128.0](https://github.com/openai/codex) 中推广并发布。我们的实现是独立的（中央 `CommandDef` 注册表、`SessionDB.state_meta` 持久化、辅助客户端评判、网关端的适配器 FIFO 延续），但理念是他们的。功劳归于应得之人。
